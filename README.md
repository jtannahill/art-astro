## How it deploys

Every push to `main` runs `.github/workflows/deploy.yml`:

1. `npm run pull-data` â€” scan DynamoDB `art-generator` table, write
   `src/data/generated/{weather,palettes}.json` (gitignored)
2. `astro check` â€” TypeScript validation across `.astro` files
3. `astro build` â€” produce `dist/` (~915 pages)
4. `pagefind --site dist` â€” build local search index
5. `aws s3 sync ./dist/ s3://art-generator-216890068001/site/`
   (no `--delete` â€” preserves pipeline-written PNG/WebP/SVG assets)
6. `aws cloudfront create-invalidation --paths "/*"`

End-to-end: ~45 s on a clean runner.

## Image pipeline

Pieces are generated by the upstream `art-weather-render` Lambda
(in [`art-generator`](https://github.com/jtannahill/art-generator)),
which writes `weather/{run_id}/{slug}/preview-2048.png` to S3.

The `art-image-resize` Lambda (S3 PUT-triggered) produces 480/960/1920
WebP variants alongside the PNG. The asset-mirror Lambda
(`art-site-rebuild`, the retired-and-shrunk legacy renderer) copies
everything from `weather/` to `site/weather/` once per daily pipeline
run so CloudFront's origin path serves the assets.

`src/components/Thumb.astro` renders `<picture>` with WebP `<source srcset>`
at three widths plus PNG fallback. Card grids load ~30 KB WebP at 480 w
instead of ~1.5 MB PNG.
